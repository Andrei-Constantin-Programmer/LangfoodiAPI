The following folders contain code, which is split into four modules representing Clean Architecture layers. You can read more about them in the Technical Report (DEL.2).

RecipeSocialMediaAPI.Domain

This is the Domain Layer of the project. You will find it also has several test projects associated with it.

The Domain Layer is the core of the API, as it contains the business logic. All entities, value objects, and domain services lie here. Validation and other important rules are defined in the Domain.

The Domain Layer has no dependencies on third-party libraries or other layers in the API, and contains pure C# code.

RecipeSocialMediaAPI.Application

This is the Application Layer of the project. You will find it also has several test projects associated with it.

The Application Layer is the orchestrator of the application's use cases. It handles requests coming from the Presentation Layer.

The Application Layer communicates directly with the Domain Layer, and has some external dependencies (MediatR, FluentValidation, BCrypt etc.)

RecipeSocialMediaAPI.Infrastructure

This is the Infrastructure Layer of the project. You will find it also has several test projects associated with it.

The Infrastructure Layer's primary purpose is data access, and handles all communication with the database (MongoDB).

The Infrastructure Layer has dependencies on both the Domain and Application layers; it implements the repository interfaces from the Application Layer.

RecipeSocialMediaAPI.Presentation

This is the Presentation Layer of the project. You will find it also has several test projects associated with it.

The Presentation Layer is the one communicating with the API consumers, and it's the birthplace of the endpoints and SignalR hubs available to them.

The Presentation Layer also configures all other layers (therefore having a dependency on all of them).
